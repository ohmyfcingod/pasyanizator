# Методология Пасьянизатора

Техническая справка по конвейеру: контракт, этапы, гейты, предохранители. Повествовательную версию
с впечатлениями см. в [ARTICLE.md](ARTICLE.md); боевые грабли и деплой-ритуал — в [ЗАРы.md](ЗАРы.md).

## Граница «данные / метаданные»

Конвейер держит жёсткую границу между тем, что требует ЖИВЫХ ДАННЫХ базы, и тем, что мутирует
МЕТАДАННЫЕ:

- **Внутрибазовая часть (1С-обработка)** — всё, что требует данных: выбор видов и реквизитов по
  реальной статистике заполненности, снятие JSON-карты, пост-накатные прогоны (права, очереди, смоук),
  миграция значений EAV → колонки. База не может добавить сама себе справочник.
- **PowerShell-конвейер снаружи** — всё, что мутирует метаданные: генерация `Catalog.<Вид>`, якорные
  врезки, гейты. Дирижёр — снаружи (накат конфы требует монопольного режима, сеанс обработки его не
  переживёт).

## Контракт: JSON-карта (vendor-neutral AST)

`contract/ПасьянсКарта.schema.json` — нормативный контракт между экстрактором и генератором. По виду:
uuid вида + имя справочника + классификатор + флаги + дефолтные поля (`тащить` по статистике) +
реквизиты (типы с квалификаторами, индексация, обязательность, статистика, предупреждения) +
`остаютсяВEAV` + `допСведения` (решение: приземлить в колонку / оставить в регистре) + `eavХвост` +
`коллизии` + `банкТипов`. Версионирование строгое: major-рассинхрон → генератор отказывается с внятной
ошибкой. Это дерево — vendor-neutral: смена пакета шаблонов `*.tpl` целит генератор в другой стек.

**Маппинг типов (ключевые случаи):** числа/строки/даты/булево — 1:1 с квалификаторами; строка(0) →
квалификатор из данных (×2 от макс, неиндексируема); неограниченная строка (LOB) — принудительно
ограничить + предупреждение (LOB нельзя индексировать и нельзя в составном типе); списочные значения
— НЕ перечисления (это данные пользователей) → внешний ключ на общий справочник значений + программный
отбор по владельцу в модуле формы; доп.сведения — приземлять в колонку по автопорогу или оставлять в
независимом регистре сведений.

## Этапы конвейера

1. **Экстрактор (в 1С)** → JSON-карта: статистика заполненности, автопороги прореживания, срез
   uuid-хвостов имён, разворот типов, детект коллизий, банк типов.
2. **G0 (Test-Карта)** — валидация карты до генерации: схема, счётные итоги, стоп-листы имён,
   коллизии с метаданными, резолвимость банка типов, квалификаторы.
3. **Генерация (New-ВидПоЭталону)** — `Catalog.<Вид>` из шаблонов: детерминированные UUID, дефолтные
   поля по флагам, списочные с FK+отбором, приземлённые сведения, предопределённый набор свойств,
   ОбработкаЗаполнения (классификатор по умолчанию).
4. **Якорная врезка (Invoke-ЯкорнаяВрезка)** — типы видов в составы ядра по свипу якорей (3 вкуса:
   ссылка/объект/владелец), с deny-list на минные паттерны.
5. **Предохранители (New-БслПредохранители)** — модуль-заглушка типов + маркер-блоки рядом с
   литеральными проверками типов в ядре.
6. **G1** — XML-валидность тронутых/созданных `.mdo`.
7. **Накат** (EDT/платформа) — реструктуризация; ритуал `clean → terminate → fullUpdate`.
8. **Гейты после наката:** Т/G5 (типозамкнутость), П (вердикт смоука), Р (регресс родного механизма).
9. **Миграция (в 1С)** — переливка значений EAV → физколонки (батч по документу, джойн физссылки,
   транзакции чанками).

## Гейты и принцип «укол-вколот»

*Гейт, который никогда не был красным, — не гейт.* Каждый гейт имеет НЕГАТИВНЫЙ тест, доказывающий, что
укол реально вколот (счёт до/после порчи) — иначе холостой укол даёт ложную веру в слепоту гейта.

| Гейт | Что проверяет | Код |
|------|---------------|-----|
| **G0** Test-Карта | контракт карты: схема, счётные итоги, стоп-листы, коллизии, резолвимость типов | `tools/Test-Карта.ps1` |
| **Т** Test-ТипоВрезка | косвенная типозамкнутость по EDT-src: матрица вид×точка врезки, маркер-блоки, парность отбора | `tools/Test-ТипоВрезка.ps1` |
| **G5** Test-ТипоЗамкнутость | ПРЯМАЯ типозамкнутость по MDClasses-экспорту: резолвимость банка типов + врезка-парность (генерённые виды обязаны иметь идентичный тип-след) | `tools/Test-ТипоЗамкнутость.ps1` |
| **П** Test-ПасьянсПрогон | вердикт сквозного смоука по файловому логу (RunId в логе И имени) | `tools/Test-ПасьянсПрогон.ps1` |
| **Чистота** Test-ЧистотаПоставки | в боевую поставку не уехал тест-харнесс (обработки, посевные виды, врезки самопрогона) | `tools/Test-ЧистотаПоставки.ps1` |

Коды возврата: 0 = зелено, 2 = красно.

## Шесть видов предохранителей (карта минных полей)

| # | Вид | Пример | Лечение |
|---|---|---|---|
| 1 | Составы типов в `.mdo` (определяемые типы, владельцы, ТЧ БП/задач) | ПредметДействия | якорная врезка |
| 2 | Литеральные `ТипЗнч(х)=Тип("…")` в коде | ветвление по виду объекта | модуль-заглушка типов + маркер-блоки |
| 3 | Явные списки типов в измерениях регистров | измерение регистра-кэша | та же якорная врезка |
| 4 | Формы с data-path через СОСТАВНОЙ реквизит | разыменование `Объект.Х.Y.Z` | deny-list (ломает ИМПОРТ конфы при зелёном EDT-гейте!) |
| 5 | Подписки, требующие РЕКВИЗИТ у объекта | префиксация → Организация | дефолтные поля вида по флагам |
| 6 | Подписки обвязки с карточным контрактом | данные документов → ДатаРегистрации | deny-list до переноса контракта |

№4 и №6 не ловятся ни EDT-гейтом, ни типозамкнутостью — только накатом и смоуком. Поэтому сквозной
смоук обязателен в каждом прогоне конвейера.

## Дисциплина конвейера

- Dry-run по умолчанию, `-Apply` осознанно; каждый прогон идемпотентен (повтор = 0 диффа).
- UUID детерминированные, сид от uuid'ов вида/свойства (не имён) — переименование не пересоздаёт колонки.
- Deny-list — только с причиной; ручные skip переживают повторный свип.
- Счётные гейты до всякого деплоя; RunId — в имени лога И в первой строке (ложное зелено от вчерашнего лога).
- Внешнее ревью — реальный гейт качества, в свежем контексте, лучше другой моделью.
