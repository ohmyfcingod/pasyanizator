# ConfigurationAssenisation — что дальше

Пасьянизатор — это **движок**. Следующая веха — обернуть его в **пак зарегистрированных скиллов
агента**, чтобы ИИ перестал быть «кодером, генерящим тексты скриптов» и стал **оператором тяжёлой
техники**: дёргает надёжные рычаги, а не ковыряет живой код 1С регулярками.

Целевой UX — одна фраза в чат:

> «Проведи ConfigurationAssenisation для вида "Заявка на оплату". Порог прореживания 2%. Мигрируй
> данные и доложи статус гейтов.»

— и конвейер отрабатывает автономно: снял карту → гейт G0 → врезал метаданные → накатил →
мигрировал → гейты Т/П → отчёт.

## Пять рычагов

| Скилл | Что делает | Ложится на |
|-------|-----------|-----------|
| `Extract-EAV-Map` | headless-1С: статистика заполненности, автопороги, срез uuid-хвостов → JSON-карта | обработка-экстрактор |
| `Run-Gate-G0` | валидация JSON-карты до генерации; при провале сам правит вход | `tools/Test-Карта.ps1` |
| `Inject-Metadata` | генерация `Catalog.<Вид>` + якорная врезка + предохранители | `tools/Invoke-Пасьянс.ps1` + `New-*.ps1` |
| `Migrate-EAV-to-SQL` | headless-1С: переливка EAV→физколонки (батч по документу, транзакции) | мигратор в смоуке |
| `Verify-Type-Closure` | гейты Т/G5 (типозамкнутость) + П (вердикт смоука) | `tools/Test-*.ps1` |

## Требования к «настоящему скиллу» (не обёртке над `.ps1`)

- **R1. Чистый CLI-контракт.** Именованные параметры, коды возврата (0=зелено, 2=красно), НОЛЬ
  интерактива и попапов. Всё, что решает человек, — параметром заранее.
- **R2. Идемпотентность.** Повторный прогон = 0 диффа. Детерминированные UUID (сид от uuid, не имён).
- **R3. Гейт на входе и выходе.** Ничего не катим красным; счётные гейты ДО деплоя; каждый гейт с
  негативным тестом (укол-вколот).
- **R4. Observability by design.** Каждый шаг оставляет артефакт на диске, читаемый агентом МИМО UI
  (карта `.json`, лог с RunId, манифест). Частичный результат переживает обрыв.
- **R5. Самокоррекция.** Агент читает машиночитаемый вывод гейта → сам правит вход → перегоняет. Гейт
  возвращает не только exit, но и адрес проблемы.
- **R6. Безопасность.** Dry-run по умолчанию, `-Apply` осознанно; деструктив — предпросмотр →
  подтверждение; снапшот базы перед батчем, откат — подмена каталога + повторный накат.
- **R7. Headless-канал для 1С-части** ⭐ (главный босс). Extract и Migrate не должны требовать
  браузерного танца с файл-флагом и открытием формы драйвера. Нужен headless-вызов обработки:
  `ibcmd` / запуск в режиме `/Execute` / внешняя обработка / тонкий HTTP-сервис внутри базы. Как
  прорубим — конвейер из «шаткого» (эмуляция браузера) станет «бронированным».
- **R8. Деплой-ритуал внутри скилла.** Правка → `clean` → `terminate` → `fullUpdate` (см.
  [ЗАРы.md](ЗАРы.md)). Обёртка над накатом обязана это соблюдать, иначе
  деплой врёт «updated» на стухшем пакете.
- **R9. Target-нейтральность.** JSON-карта = vendor-neutral AST; смена пакета шаблонов `*.tpl`
  (`Catalog.mdo` → `schema.prisma` / DocType / …) целит генератор в другой стек. Держать схему и
  решения нейтральными, где дёшево.

## Состояние

«Мозг» готов и обкатан: контракт JSON-карты, гейты (G0/Т/G5/П) с негативами, генератор с
детерминированными UUID и якорной врезкой — всё в этом репозитории. «Руки, которые лезут в живую 1С»
(Extract, Migrate) доказаны на стенде цифра-в-цифру, но пока вызываются через браузер+файл-флаг.
До «рук базуки» — три вещи, ни одна не про переписывание движка:

1. **R7 — headless-канал 1С** (главное): вынести Extract/Migrate в headless-вызов.
2. **Регистрация**: на каждый рычаг — манифест скилла; чистые PS-гейты заворачиваются почти как есть.
3. **Оркестратор**: `Invoke-ConfigurationAssenisation -Вид X -Порог Y -Apply -Migrate`, гоняющий E2E
   и докладывающий статус гейтов машиночитаемо.

Родственная идея — **Приземлятор ДопРеквизитов**: тот же EAV→физика, но генерализованный на любую
конфигурацию на БСП (не только Документооборот). Оба хотят один headless-канал и один реестр скиллов.
