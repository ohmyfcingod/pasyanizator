# ЗАРы — энциклопедия выживания в Enterprise 1С

**ЗАР** = **З**аписанное **А**рхитектурное **Р**ешение (inline-комментарий рядом с кодом, объясняющий
«почему так и почему нельзя трогать» — Контекст → Ограничение → Решение, вне времени). Здесь —
курированный свод боевых грабель проекта. Каждая оплачена сожжёнными нервами и часами дебага; такое
не нагуглить на форумах — только собственный опыт превращает боль в защиту.

## Зелёный гейт — рантайм-краш (семья из пяти реинкарнаций)

Класс багов, который проходит проверку 0 ошибок EDT, но падает в рантайме:

- **Клиентский тип/объект в СЕРВЕРНОМ модуле** (`ХранилищеСистемныхНастроек` и т.п.): синтакс-чек
  контексты НЕ различает (зелёный!), а в рантайме модуль обработки НЕ КОМПИЛИРУЕТСЯ → создание
  обработки рушит ВЕСЬ серверный вызов, форма не создаётся, `Попытка` бессильна, в логах ПУСТО.
- **Имена локальных переменных, совпавшие** с реквизитами формы, встроенными свойствами формы
  (`Параметры`, `Элементы`), ОБЩИМИ МОДУЛЯМИ конфигурации, глобальными функциями (`Строка`) —
  молчаливая перепривязка, падение «значение не является значением объектного типа». Мёртвые
  реквизиты — удалять: это мина.
- **Формы с data-path через составной реквизит** (`Объект.Обращения.Обращение.Контрагент`): вставка
  типа без этих полей ломает ИМПОРТ конфигурации — при зелёном EDT-гейте! Ловится только импортом.

**Вывод:** после правки серверных функций модуля формы обязателен клик по реальной команде в
веб-клиенте. Гейт и рендер проверяют синтаксис и вёрстку, но не выполнение.

## Платформа и запросы

- **`СтрШаблон` дерётся с литеральным `%`** в `ПОДОБНО "…%"` — принимает процент за маркер параметра,
  падает «ошибка синтаксиса шаблона». Имя таблицы/поля подставлять КОНКАТЕНАЦИЕЙ, не шаблоном.
  (Коварно: если запрос выполняется ВНЕ `Попытки`, исключение улетает мимо лога — искать в логе выше.)
- **SDBL, длина строки:** ограниченная ≤ 1024; неограниченная (LOB) — нельзя индексировать и нельзя
  в составном типе. Экстрактор режет длину под это (не «масштабирование врезки в составные», как
  подумали сначала, — это диагностировал дед).
- **Иерархический справочник:** поля запроса имеют тип «Т + Null» (NULL у групп) — детект типов обязан
  игнорировать Null. И у справочника может НЕ быть Кода — состав колонок проверять по факту, не по вере.
- **Регистр сведений с независимой записью:** писать МЕНЕДЖЕРОМ записи по ПОЛНОМУ ключу, НЕ набором
  (набор без отбора снёс бы весь глобальный регистр). У менеджера записи независимого РС нет
  `ОбменДанными` (это у наборов).
- **`ВладелецДополнительныхЗначений` руками ставить нельзя** — БСП сочтёт свойство «созданным по
  образцу» и запретит создавать значения; self-reference проставляет сама подписка при первом значении.
- **Предопределённый набор свойств** `Тип_Имя`: БСП зовёт `НаборыДополнительныхРеквизитовИСведений[имя]`
  где имя = `СтрЗаменить("Справочник.Х", ".", "_")`; у сгенерённого объекта его нет → падение
  «поле объекта не обнаружено». Лечение: генератор врезает пустой предопределённый набор.

## Массовые операции (ETL-масштаб)

- **Батчить по СУЩНОСТИ, а не по значению.** Перенос N значений одного документа = ОДНА запись, не N
  (иначе на 100k×5 — N поисков+чтений+записей на документ → IPC/IOPS-шторм, взрыв transaction log).
  Паттерн: упорядочить по ключу → накапливать значения в объект → записать один раз на смене ключа;
  батч-фиксация транзакции считает документы, а не значения.
- **Физссылку — джойном в запросе**, не поиском по имени в цикле (поиск по неиндексированной строке × N).
- **Транзакция чанками:** начать → пачка записей (≤1000) → зафиксировать → следующая; на исключении
  откатить. Одна транзакция на 5М взорвёт лог; без транзакции — IOPS-затык на автотранзакции каждой записи.
- **Чтение Excel:** НЕ обход ячеек через `COMОбъект("Excel.Application")` — 1,5М кросс-процессных
  вызовов на 100к строк роняют `rphost`. Нативно: `ТабличныйДокумент.Прочитать()` (сам распаковывает
  zip+xml на C++), либо `ТабДок` + `ПостроительОтчёта.Результат.Выгрузить()`, либо буфер обмена + Ctrl+V.

## Ритуал деплоя автономного сервера

Кэш метаданных сервера 1С не подчиняется законам физики: в памяти может крутиться старый бинарный
кэш, на клиенте новый интерфейс, а в базе третья версия кода. Единственный доказанный способ
загрузить ровно ту конфигурацию, что скормили:

1. **`clean_project`** после ручной правки (НЕ облегчённая ревалидация — она однажды протащила стухший
   скомпилированный модуль: деплой врал «updated», а в базе крутился старый код).
2. **Терминировать сеансы ПЕРЕД накатом** — не форсить в живые (закрытие вкладки браузера сеанс не
   отцепляет, он висит ~5 мин; форс множит зомби-сеансы и лок монопольного режима).
3. **После `clean` — полный накат**, не инкремент (инкрементальный пакет после clean может тащить мусор).

## Гейты и ревью

- **Негатив обязан доказать, что укол ВКОЛОТ** (счёт до/после порчи): холостой укол даёт ложную веру
  в слепоту гейта. Поймано на гейте типозамкнутости: замена искала CRLF в LF-файле, файл не менялся,
  «негатив» позеленел — гейт был ни при чём.
- **Внешнее ревью — реальный гейт.** Свежая независимая модель с «критикуй беспощадно» ловит слепые
  пятна: баг потери данных в подчинённых регистрах, write-amplification миграции, слепую зону гейта на
  списочных сведениях. Под каждый must-fix — постоянный сборочный тест (фикс без теста воскресает).
- **Красный гейт прав, пока не доказано обратное.** Синтакс-чекер трижды блокировал «валидный» код —
  форс обхода дал те же ошибки от платформы. Обход гейта — только поняв, ПОЧЕМУ он красный.
