# Пасьянизатор — быстрый старт

**Пасьянизатор** декомпозирует EAV-монолит дополнительных реквизитов и сведений (механизм свойств БСП)
в физические `Справочник.<Вид>` со статическими SQL-колонками. Конвейер:
**снять карту → проверить → сгенерировать → накатить → пост-проверки.**

Карта (`ПасьянсКарта_*.json`, `schemaVersion 1.2`) — vendor-neutral описание модели: виды, реквизиты,
типы, статистика заполненности. Генератор материализует её в EDT-исходники через набор шаблонов `*.tpl`.

## Что работает «из коробки», а что — образец

- **Снять карту вашей базы и проверить её (шаги 1–2) — turnkey.** Экстрактор работает на любой
  конфигурации на БСП с механизмом свойств; из коробки запросная часть заточена под
  **1С:Документооборот 3.x** (для другой конфы её адаптируют). За пару минут вы получаете JSON-карту
  EAV-модели своей базы.
- **Генерация физики, врезка типов, накат, миграция (шаги 3–6) — референс-реализация под ДО3.**
  Шаблоны `*.tpl` и карта якорей `tools/КартаВрезок.tsv` заточены под структуру ДО3. Для другой
  конфигурации это рабочий образец: карта vendor-neutral, целевой стек задаётся набором шаблонов.

## 0. Что нужно

- **1С:Предприятие 8.3+**, конфигурация на **БСП** с **механизмом свойств** (доп. реквизиты и сведения).
- **PowerShell 7+ (`pwsh`)** — для генератора и гейтов (запущенная 1С не требуется).
- **EDT или Конфигуратор** — чтобы накатить сгенерированные метаданные на базу.

> В значениях параметров по умолчанию скрипты содержат пути окружения разработки — всегда указывайте
> свои через аргументы (`-Карта`, `-Src` и т. д.).

## 1. Снять карту

**С формой (проще всего).** Соберите/подключите внешнюю обработку `ПасьянизаторЭкстрактор`
(EDT-исходники и инструкция — в [`tools/extractor/`](tools/extractor/)). Форма-витрина покажет виды
документов с числом реквизитов и документов; отметьте нужные (или задайте порог), укажите каталог и
нажмите **«Сформировать карту»** — получите JSON-файл.

![Витрина экстрактора: обзор видов, выбор состава, каталог выгрузки](pictures/Экстрактор.png)

**Кодом.** Вызовите обработку из встроенного языка; возврат — `Соответствие`, значения читаются по
ключу `["…"]`:

```bsl
Параметры = Новый Структура;
Параметры.Вставить("ИменаВидов", Новый Массив);            // пусто = автоподбор топ-N по числу доп. реквизитов
Параметры.Вставить("МаксВидов", 3);
Параметры.Вставить("ПорогПрореживанияПроцент", 1);
Параметры.Вставить("КаталогВыгрузки", "C:\Пасьянс\Карты"); // ваш каталог
Результат = ВнешниеОбработки.Создать(Путь, Ложь).СформироватьКарту(Параметры);
Сообщить(Результат["сообщение"]);   // "Карта: видов N, реквизитов M, предупреждений K"
// Результат["успех"] (Булево), Результат["файл"] (путь к ПасьянсКарта_<База>_<дата>.json)
```

## 2. Проверить карту (гейт G0)

```powershell
pwsh -NoProfile -File tools/Test-Карта.ps1 `
  -Карта "C:\Пасьянс\Карты\ПасьянсКарта_<База>_<дата>.json" `
  -Src   "<корень EDT-исходников вашей конфигурации>"
# exit 0 = карта в порядке, 2 = ошибки (валидация / JSON не парсится)
```

G0 проверяет схему и строгую версию `schemaVersion`, счётные итоги, коллизии имён с метаданными базы,
покрытие и квалификаторы типов. Прогонять карту через G0 стоит **до** генерации.

## 3. Сгенерировать `Catalog.<Вид>`

По умолчанию — **dry-run** (ничего не пишет, только показывает план):

```powershell
pwsh -NoProfile -File tools/Invoke-Пасьянс.ps1 `
  -Карта "...\ПасьянсКарта_<База>_<дата>.json" `
  -Виды  Договор,ГарантийныйДоговор `
  -Src   "<корень EDT-исходников>"
```

Запись — добавьте `-Apply`. Оркестратор идёт строго по этапам, любой провал останавливает прогон
(`exit 2`): пред-валидация (`-Виды` должны быть в карте) → G0 → генерация справочников из шаблонов →
якорная врезка типов в составы ядра → предохранители → проверка XML-валидности. Прогон **идемпотентен**
(повтор = 0 диффа); UUID детерминированные, поэтому переименование не пересоздаёт колонки, а платформа
делает `RENAME`, а не `DROP`.

> Отдельный вид — `tools/New-ВидПоЭталону.ps1`. Врезка типов идёт по `tools/КартаВрезок.tsv` (карта
> якорей под ДО3); для своей конфигурации её пересобирают через `tools/Find-ЯкорныеТочки.ps1`.

## 4. Накатить изменения на базу

Сгенерированные объекты попадают в **EDT-исходники** вашей конфигурации. Накатите их штатно:

- **EDT:** синхронизируйте проект и выполните **«Обновить конфигурацию базы данных»** (реструктуризация).
- **Конфигуратор:** «Загрузить конфигурацию из файлов» → **«Обновить конфигурацию базы данных»**.

Обновление требует монопольного доступа — завершите активные сеансы. Платформа перестроит таблицы под
новые физические реквизиты. Типовые грабли реструктуризации разобраны в [docs/ЗАРы.md](docs/ЗАРы.md).

## 5. Пересчитать права, кэши и индексы

После добавления новых объектов метаданных часть агрегатов 1С **не пересчитывается сама**. Надёжный
и версионно-независимый способ — штатные панели **Начальная страница → «Поддержка и обслуживание»**:

- **Управление расчётом прав** — пересчёт прав доступа/RLS для новых видов.
- **Результаты обновления и дополнительная обработка данных** — прогон отложенных обработчиков
  обновления БСП (закрывают очередь кэшей и отложенные пересчёты).
- **Управление полнотекстовым поиском** — добавить новые справочники в индекс, если пользуетесь ФТС.

Программные эквиваленты (`РегистрыСведений.ПраваРолей.ОбновитьДанныеРегистра()`, обработка очереди
кэширующих данных) и версионные нюансы БСП — в [docs/ЗАРы.md](docs/ЗАРы.md).

## 6. Пост-накатные проверки (гейты)

**Типозамкнутость по MDClasses-экспорту** (G5) — самая полезная проверка: все сгенерированные виды
обязаны иметь идентичный тип-след во всех составах ядра. Нужен designer-XML экспорт конфигурации:

```powershell
pwsh -NoProfile -File tools/Test-ТипоЗамкнутость.ps1 -Экспорт "C:\Пасьянс\export-xml" -Карта "...\карта.json"
# 0 = типозамкнуто, 2 = висячие типы / расхождение, 1 = ошибка входа (нет каталога/карты)
```

**Косвенная проверка по EDT-исходникам** (гейт Т) и **чистота поставки** (тестовые объекты не уехали
в боевую конфигурацию):

```powershell
pwsh -NoProfile -File tools/Test-ТипоВрезка.ps1     -Src "<корень EDT-исходников>" -Карта "...\карта.json"
pwsh -NoProfile -File tools/Test-ЧистотаПоставки.ps1 -Src "<корень EDT-исходников>"
```

Коды возврата у всех гейтов: `0` = зелено, `2` = красно (у части — ещё `1` = ошибка входа).
Финальная проверка — открыть сгенерированный вид в базе и провести его штатным процессом
(в ДО3: создать документ → отправить на согласование → убедиться, что задача уходит исполнителю).

## Что дальше / target-нейтральность

Карта — это **vendor-neutral AST** модели, независимый от целевой платформы. Генератор материализует
её в 1С через набор шаблонов `*.tpl` (`tools/Шаблоны/`). **Смена набора шаблонов целит ту же карту в
другой стек** — вы строите не генератор `.mdo`, а компилятор бизнес-логики. Разбор подхода —
[docs/Методология.md](docs/Методология.md), история проекта — [docs/ARTICLE.md](docs/ARTICLE.md).

## Граница

Публикуется только **наш код** (лицензия MIT): экстрактор, гейты, генератор, шаблоны и контракт карты.
Конфигурация-донор **1С:Документооборот 3.x — проприетарный продукт «1С»** и в поставку не входит;
сгенерированные `Catalog.<Вид>` — производные от неё и тоже не публикуются.
